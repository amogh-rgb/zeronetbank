6. APP SERVICES & STATE MANAGEMENT
================================================================================

CORE SERVICES:
─────────────────────────────────────────────────────────────────────────────

1. API SERVICE (api_service.dart)
   Purpose: Handle all HTTP communication with backend
   Base URL: https://zeronetpaybank.onrender.com/api
   
   Methods:
   - get(String endpoint) → Future<Response>
   - post(String endpoint, dynamic body) → Future<Response>
   - put(String endpoint, dynamic body) → Future<Response>
   - delete(String endpoint) → Future<Response>
   
   Features:
   - Automatic JWT token attachment
   - Request/Response interceptors
   - Error handling
   - Timeout configuration
   - Retry logic

2. WALLET SERVICE (wallet_service.dart)
   Purpose: Manage wallet operations
   
   Methods:
   - getBalance() → Future<double>
   - getWalletInfo() → Future<Wallet>
   - sendMoney(SendRequest) → Future<Transaction>
   - receiveMoney(String amount) → Future<String>
   - getTransactionHistory() → Future<List<Transaction>>
   
3. USER SERVICE (user_service.dart)
   Purpose: User account management
   
   Methods:
   - login(String email, String password) → Future<User>
   - register(User user) → Future<User>
   - getProfile() → Future<User>
   - updateProfile(User user) → Future<User>
   - changePassword(String old, String new) → Future<bool>
   - logout() → Future<void>

4. AUTHENTICATION SERVICE (auth_service.dart)
   Purpose: Handle all auth flows
   
   Methods:
   - authenticateWithBiometric() → Future<bool>
   - authenticateWithPin(String pin) → Future<bool>
   - savePin(String pin) → Future<void>
   - verifyPin(String pin) → Future<bool>
   - getStoredToken() → Future<String?>
   - saveToken(String token) → Future<void>
   - clearSession() → Future<void>

5. QUEUE SERVICE (queue_service.dart)
   Purpose: Monitor and manage queue issues
   
   Methods:
   - getQueueIssues() → Future<List<QueueIssue>>
   - retryTransaction(String id) → Future<bool>
   - getQueueStatus() → Future<QueueStatus>

STATE MANAGEMENT (Riverpod):
─────────────────────────────────────────────────────────────────────────────

Providers:

1. authProvider (StateNotifier)
   State: AuthState (initial, loading, authenticated, error)
   Actions: login, logout, checkAuth

2. userProvider (FutureProvider)
   State: AsyncValue<User>
   Actions: fetchProfile, updateProfile

3. walletProvider (StateNotifier)
   State: WalletState (balance, transactions, loading)
   Actions: loadWallet, refreshBalance, sendMoney

4. transactionsProvider (FutureProvider)
   State: AsyncValue<List<Transaction>>
   Actions: loadTransactions, filter, search

5. queueProvider (StreamProvider)
   State: AsyncValue<List<QueueIssue>>
   Actions: monitorQueue, retryFailed

9. DATABASE SCHEMA (PostgreSQL)
================================================================================

TABLE: users
─────────────────────────────────────────────────────────────────────────────
Purpose: Store user accounts

Columns:
- id: String (PK, CUID) - Unique identifier
- email: String (Unique) - User email address
- phone: String (Unique) - Mobile number
- password: String? - Hashed password (optional)
- displayName: String? - Full name
- publicKey: String - Crypto public key
- balance: Decimal (20,8) - Account balance
- createdAt: DateTime - Account creation
- updatedAt: DateTime - Last update

Relations:
- One-to-Many: users → transactions

Sample Record:
{
  id: "clxyz123abc",
  email: "john.doe@example.com",
  phone: "+1234567890",
  displayName: "John Doe",
  balance: 2500.50
}

TABLE: transactions
─────────────────────────────────────────────────────────────────────────────
Purpose: Store all financial transactions

Columns:
- id: String (PK, CUID)
- from: String - Sender email
- to: String - Receiver email
- amount: Decimal (20,8) - Transaction amount
- type: String - DEPOSIT, TRANSFER, WITHDRAWAL, PAYMENT
- status: String - COMPLETED, PENDING, FAILED
- signature: String? - Digital signature
- description: String? - Transaction note
- timestamp: DateTime - When occurred
- userId: String (FK) - Link to users.id

Relations:
- Many-to-One: transactions → users

Sample Record:
{
  id: "clabc456xyz",
  from: "john@example.com",
  to: "jane@example.com",
  amount: 250.00,
  type: "TRANSFER",
  status: "COMPLETED",
  description: "Payment for services"
}

API SERVICE → Adds JWT Token (if authenticated)
    ↓
Dio Client → Sends to Backend URL
    ↓
NODE.JS SERVER → Receives Request
    ↓
Express Router → Routes to Handler
    ↓
Controller/Service → Business Logic
    ↓
Prisma ORM → SQL Query
    ↓
PostgreSQL → Returns Data
    ↓
Prisma → Maps to Objects
    ↓
Service → Formats Response
    ↓
Express → JSON Response
    ↓
Dio Client → Receives JSON
    ↓
Flutter App → Updates UI

API CONFIGURATION:
─────────────────────────────────────────────────────────────────────────────

Base URL: https://zeronetpaybank.onrender.com/api

Headers:
- Content-Type: application/json
- Authorization: Bearer {JWT_TOKEN} (when authenticated)

Timeout: 30 seconds
Retry: 3 attempts with exponential backoff

================================================================================

14. AUTHENTICATION FLOW
================================================================================

INITIAL LOGIN:
─────────────────────────────────────────────────────────────────────────────

1. USER enters credentials in Flutter app
2. APP sends POST /api/users/login
3. BACKEND validates credentials
4. BACKEND generates JWT token
5. BACKEND returns { user, token }
6. APP stores token in SecureStorage
7. APP navigates to Dashboard

SUBSEQUENT REQUESTS:
─────────────────────────────────────────────────────────────────────────────

1. APP makes API request
2. API SERVICE retrieves token from storage
3. ADDS header: Authorization: Bearer {token}
4. SENDS request to backend
5. BACKEND verifies JWT signature
6. BACKEND extracts user ID from token
7. BACKEND processes request
8. BACKEND returns response

TOKEN REFRESH:
─────────────────────────────────────────────────────────────────────────────

- Token expires after 24 hours
- App detects 401 Unauthorized
- Redirects to login screen
- User re-authenticates
- New token issued

BIOMETRIC AUTH (App Only):
─────────────────────────────────────────────────────────────────────────────

1. User enables biometric in settings
2. App stores credential in secure storage
3. Next login: Prompt for fingerprint/face
4. If biometric valid → Auto-login with stored token
5. If biometric fails → Fallback to PIN/Password

PIN AUTH (App Only):
─────────────────────────────────────────────────────────────────────────────

1. User sets PIN during registration
2. PIN stored hashed in secure storage
3. Login options: PIN or Password
4. PIN verification uses constant-time comparison
5. After 3 failed attempts → Lockout

19. TROUBLESHOOTING GUIDE
================================================================================

FLUTTER APP ISSUES:
─────────────────────────────────────────────────────────────────────────────

App won't build:
→ flutter clean
→ flutter pub get
→ flutter build apk

API connection failed:
→ Check internet connection
→ Verify API_BASE_URL in .env
→ Check backend is running
→ Check CORS settings

Login not working:
→ Clear app data (Settings → Apps → Clear Data)
→ Reinstall app
→ Check credentials

Biometric not working:
→ Check device supports biometric
→ Enable in device settings
→ Re-register in app

Backend connection refused:
→ Verify not using http://localhost on real device
→ Use actual deployed URL
→ Check firewall settings

BACKEND ISSUES:
─────────────────────────────────────────────────────────────────────────────

Build fails:
→ npm install
→ npx prisma generate
→ npm run build

Database connection error:
→ Check DATABASE_URL format
→ Verify database exists
→ Check network access

API returning 500:
→ Check server logs
→ Verify Prisma client generated
→ Check database schema

CSP errors in browser:
→ Normal - we're using inline SVGs
→ Should not block functionality

Dashboard shows zeros:
→ Database is empty
→ Run: npx prisma db seed
→ Or manually add users
