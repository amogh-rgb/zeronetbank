                         5. DATABASE SCHEMA (Detailed)
================================================================================

DATABASE NAME: zeronetbank_db
PROVIDER: PostgreSQL

------------------------------------------------------------------------------
TABLE: users
------------------------------------------------------------------------------
Purpose: Store user account information

Columns:
- id          : String (Primary Key, Auto-generated CUID)
- email       : String (Unique, Required)
- phone       : String (Unique, Required) - Mobile number
- password    : String (Optional, hashed)
- displayName : String (User's full name)
- publicKey   : String (For crypto features)
- balance     : Decimal (20,8 precision) - Default: 1000.00
- createdAt   : DateTime - Auto-set on creation
- updatedAt   : DateTime - Auto-updated on modify

Indexes:
- PRIMARY KEY (id)
- UNIQUE INDEX (email)
- UNIQUE INDEX (phone)

Relationships:
- One-to-Many with transactions table
- A user can have multiple transactions

Sample Data:
id: "clxyz123abc", 
email: "john.doe@example.com",
phone: "+1234567890",
displayName: "John Doe",
balance: 2500.50

------------------------------------------------------------------------------
TABLE: transactions
------------------------------------------------------------------------------
Purpose: Store all financial transactions

Columns:
- id          : String (Primary Key, Auto-generated CUID)
- from        : String (Sender email or identifier)
- to          : String (Receiver email or identifier)
- amount      : Decimal (20,8 precision) - Transaction amount
- type        : String - Transaction category:
                * DEPOSIT    - Money added to account
                * TRANSFER   - Money sent to another user
                * WITHDRAWAL - Money taken out
                * PAYMENT    - Purchase/payment made
- status      : String - Current state:
                * COMPLETED - Successfully processed
                * PENDING   - In progress
                * FAILED    - Error occurred
- signature   : String (Optional) - Digital signature
- description : String (Optional) - Transaction details
- timestamp   : DateTime - When transaction occurred
- userId      : String (Foreign Key) - Links to users.id

Indexes:
- PRIMARY KEY (id)
- INDEX (userId) - For quick user transaction lookup
- INDEX (timestamp) - For chronological sorting

Relationships:
- Many-to-One with users table
- Each transaction belongs to one user

Sample Data:
id: "clabc456xyz",
from: "john.doe@example.com",
to: "jane.smith@example.com",
amount: 250.00,
type: "TRANSFER",
status: "COMPLETED",
description: "Payment for services",
userId: "clxyz123abc"

                         6. API ENDPOINTS (Complete)
================================================================================

BASE URL: https://zeronetpaybank.onrender.com/api

------------------------------------------------------------------------------
ADMIN ENDPOINTS (/api/admin/*)
------------------------------------------------------------------------------

1. GET /api/admin/dashboard
   Description: Get dashboard statistics
   Response: {
     success: true,
     stats: {
       totalUsers: 3,
       totalTransactions: 10,
       totalBalance: "7501.25",
       activeUsers: 3
     }
   }

2. GET /api/admin/users
   Description: Get list of all users
   Response: {
     success: true,
     users: [
       {
         id: "...",
         email: "...",
         displayName: "...",
         phone: "...",
         balance: "...",
         transactionCount: 5
       }
     ]
   }

3. GET /api/admin/transactions
   Description: Get all transactions
   Response: {
     success: true,
     transactions: [...]
   }

------------------------------------------------------------------------------
USER ENDPOINTS (/api/users/*)
------------------------------------------------------------------------------

1. POST /api/users/register
   Description: Create new user account
   Body: {
     name: "John Doe",
     email: "john@example.com",
     mobile: "+1234567890",
     password: "hashedpassword"
   }
   Response: {
     success: true,
     message: "User registered successfully",
     user: { id, email, name, mobile, balance }
   }

2. POST /api/users/login
   Description: Authenticate existing user
   Body: { email, password }
   Response: { success, message, user }

3. GET /api/users/profile/:email
   Description: Get user profile with transactions
   Response: {
     success: true,
     user: {
       id, email, name, mobile, balance,
       transactions: [...]
     }
   }

4. GET /api/users/transactions/:email
   Description: Get user's transaction history
   Response: {
     success: true,
     transactions: [
       {
         id, from, to, amount, type,
         status, description, timestamp
       }
     ]
   }

5. GET /api/users/by-phone/:phone
   Description: Find user by mobile number
   Response: {
     success: true,
     user: { id, email, name, phone, balance }
   }

------------------------------------------------------------------------------
OTP ENDPOINTS (/api/otp/*) [Optional Feature]
------------------------------------------------------------------------------

1. POST /api/otp/send
   Body: { identifier, type }
   Response: { success, message, otp }

2. POST /api/otp/verify
   Body: { identifier, otp }
   Response: { success, message, type }

3. GET /api/otp/current/:identifier
   Response: { success, otp, expires, type }

                         10. SECURITY FEATURES
================================================================================

IMPLEMENTED SECURITY:

1. CONTENT SECURITY POLICY (CSP)
   Header: Content-Security-Policy: script-src 'self' 'unsafe-inline'
   - Blocks external scripts from unknown sources
   - Prevents XSS attacks
   - Only allows same-origin JavaScript

2. NO INLINE EVENT HANDLERS
   - Removed all onclick="function()" attributes
   - Using addEventListener() in external JS files
   - Prevents script injection

3. EXTERNAL CDN BLOCKING
   - No Font Awesome CDN
   - No external CSS/JS libraries
   - All icons are inline SVG
   - All code is self-hosted

4. HELMET.JS MIDDLEWARE
   - Sets security headers automatically
   - X-Content-Type-Options: nosniff
   - X-Frame-Options: DENY
   - X-XSS-Protection: 1; mode=block

5. CORS PROTECTION
   - Configured for specific origins
   - Prevents unauthorized API access
   - Credentials handling secured

6. PRISMA ORM SECURITY
   - Automatic SQL injection prevention
   - Parameterized queries
   - Input validation

7. NO PASSWORD STORAGE (Demo Mode)
   - Simplified authentication
   - Mobile-based login
   - No sensitive credential storage

                         14. TROUBLESHOOTING
================================================================================

PROBLEM: Admin login button not working
SOLUTION: 
- Clear browser cache (Ctrl+Shift+R)
- Check browser console for errors
- Ensure JavaScript is enabled
- Try incognito/private mode

PROBLEM: User mobile not found
SOLUTION:
- Use test numbers: +1234567890, +0987654321, +1122334455
- Or enter any new number to register

PROBLEM: Dashboard shows 0 users/transactions
SOLUTION:
- Database may be empty
- Run seed script to add sample data
- Check server logs for errors

PROBLEM: CSP errors in console
SOLUTION:
- Already fixed - ensure using latest files
- No external CDN resources should load
- All scripts should be from 'self'

PROBLEM: Page not loading
SOLUTION:
- Check internet connection
- Verify URL: https://zeronetpaybank.onrender.com
- Wait for Render deployment to complete (2-3 mins)
- Check Render dashboard for service status

PROBLEM: API errors
SOLUTION:
- Check browser Network tab
- Verify API endpoint format
- Check server logs on Render
- Ensure DATABASE_URL is set correctly
